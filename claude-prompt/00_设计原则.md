# 提示词设计原则

> 好的提示词不是技巧堆砌，而是理解模型的认知模式

---

## 原则一：结构化优于堆砌

好的提示词不是形容词的堆砌，而是清晰的思考框架。用具体的步骤和约束替代模糊描述。

### ❌ 错误示例
```
你是最厉害的专家，给我最好最全面最权威的答案
```

### ✅ 正确示例
```
你是[领域]专家，请按以下步骤分析：
1. 先梳理背景事实
2. 再分析核心问题
3. 最后给出可执行建议
```

---

## 原则二：角色定位要具体

"你是专家"不如"你是一位审稿人"，后者隐含了批判性、严谨性、关注方法论等特质。

### 角色具体化对照表

| 模糊角色 | 具体角色 | 隐含特质 |
|---------|---------|---------|
| 法律专家 | 劳动纠纷律师 | 熟悉劳动法、仲裁流程、赔偿计算 |
| 写作专家 | SCI论文审稿人 | 学术规范、方法论严谨、引用要求 |
| 技术专家 | 系统架构师 | 全局视角、技术选型、可扩展性 |
| 编程专家 | 全栈工程师 | 前后端、数据库、部署运维 |

---

## 原则三：约束即自由

明确的限制反而让模型输出更聚焦。模糊的开放性请求往往得到泛泛的回答。

### 有效约束类型

```markdown
# 格式约束
- 使用表格对比选项
- 每段不少于5句话
- 代码必须有注释

# 内容约束
- 必须引用具体法条/论文
- 禁止编造数据和案例
- 计算精确到小数点后两位

# 范围约束
- 核心创新点不超过3个
- 只分析最近2年的技术
- 聚焦于[具体领域]
```

---

## 原则四：引用与验证

对于专业领域，要求模型引用具体来源并使用工具验证，可以大幅提升准确性。

### 验证要求模板

```markdown
# 工作原则
1. 所有结论必须有依据支撑（引用来源/数据/案例）
2. 不确定的内容如实说明，不编造
3. 可使用搜索工具验证和更新知识
4. 对于引用的内容，标注具体来源以便查证
```

---

## 原则五：迭代优化

第一轮输出后，可以追问深化：

```
- "展开第X点，给出更详细的实现路径"
- "作为批评者，你会对这个方案提出哪些质疑？"
- "请严格阅读材料，重新梳理思路后回答"
- "请review后优化出最佳答案"
```

---

## 常见错误避免

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| "最好的""最全面的"等模糊形容词 | 给出具体的评判标准和约束 |
| 一次性提出过多要求 | 分步骤、分模块提出 |
| 没有明确输出格式 | 指定结构、字数、格式要求 |
| 忽略验证和引用要求 | 要求引用来源、使用搜索验证 |
| 角色定位过于宽泛 | 使用具体职业角色激活特定能力 |

---

## Few-shot 示例技巧

好的示例不只是"正确答案"，而是展示你想要的：
- **推理路径** - 如何一步步得出结论
- **输出格式** - 表格、列表、段落的使用
- **详略程度** - 哪些要展开，哪些要简略

> 💡 一个好例子 + 一个反例，往往比三个好例子更有效
