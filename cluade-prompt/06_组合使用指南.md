# 模板组合使用指南

> 复杂任务往往需要组合多个模板

---

## 组合原则

### 1. 分层组合
```
基础模板（角色+原则）
    ↓
领域模板（专业要求）
    ↓
任务模板（具体输出）
```

### 2. 串联组合
```
任务1输出 → 任务2输入 → 任务3输入 → 最终输出
```

### 3. 并联组合
```
任务A ──┐
任务B ──┼──→ 汇总整合
任务C ──┘
```

## 常见组合场景

### 场景1：学术论文 + 代码实现

**适用情况**：需要根据代码写论文，或根据论文实现代码

**组合方式**：
```
学术研究模板 + 编程开发模板
```

**流程**：
1. 用学术模板写方法论章节（描述算法原理）
2. 用编程模板实现核心代码
3. 用学术模板写实验分析
4. 整合为完整论文

**示例提示词**：
```markdown
# 第一步：写方法论
[使用学术研究模板]
根据以下代码，撰写论文的Methodology部分...

# 第二步：补充代码
[使用编程开发模板]
根据上述方法论，补充实验代码和可视化脚本...

# 第三步：写实验分析
[使用学术研究模板]
根据实验结果，撰写Results and Analysis部分...
```

### 场景2：法律分析 + 文档生成

**适用情况**：需要法律咨询并生成正式法律文书

**组合方式**：
```
法律咨询模板 + 文档输出要求
```

**流程**：
1. 用法律模板进行完整分析（引用法条、计算金额）
2. 基于分析结果生成法律文书
3. 确保文书格式符合规范

**示例提示词**：
```markdown
# 第一步：法律分析
[使用法律咨询模板]
请分析我的劳动纠纷案件...

# 第二步：生成文书
基于上述分析，请帮我生成劳动仲裁申请书，要求：
1. 符合仲裁委的格式规范
2. 事实和理由部分引用具体法条
3. 请求事项金额与计算结果一致
4. 保持逻辑清晰、论证有力
5. 输出Word格式文档
```

### 场景3：技术调研 + 架构设计

**适用情况**：需要调研技术现状并设计新方案

**组合方式**：
```
技术架构模板 + 搜索工具调用
```

**流程**：
1. 使用搜索工具调研最新技术动态
2. 按照架构模板的四阶段框架分析
3. 输出架构设计文档

**示例提示词**：
```markdown
# 角色
你是系统架构师

# 任务
为[领域]设计新一代架构方案

# 工作方式
1. 先使用搜索工具调研：
   - 当前主流方案
   - 近2年的新技术/论文
   - 开源项目和最佳实践

2. 按四阶段框架分析：
   - 现状解构
   - 跨域迁移
   - 重组创新
   - 验证边界

3. 输出完整架构设计文档
```

### 场景4：需求分析 + 原型开发

**适用情况**：从需求到可演示的原型

**组合方式**：
```
通用基础模板（需求分析）+ 编程开发模板（原型实现）
```

**流程**：
1. 理解和拆解业务需求
2. 设计数据模型和交互流程
3. 实现前端原型Demo

**示例提示词**：
```markdown
# 第一步：需求分析
[使用通用基础模板]
我需要做一个[项目]，请帮我：
1. 梳理核心功能点
2. 设计数据模型
3. 规划交互流程

# 第二步：原型开发
[使用编程开发模板]
基于上述设计，实现一个前端Demo，要求：
- 单HTML文件
- 完整的UI交互
- 模拟数据流
```

### 场景5：问题诊断 + 解决方案

**适用情况**：遇到技术问题需要诊断和解决

**组合方式**：
```
Bug修复模板 + 代码优化模板
```

**流程**：
1. 描述问题现象和错误信息
2. 分析问题根因
3. 提供修复方案和优化建议

**示例提示词**：
```markdown
# 第一步：问题诊断
[使用Bug修复模板]
问题描述：[...]
错误信息：[...]
相关代码：[...]

# 第二步：优化建议
除了修复这个bug，请顺便审查这段代码：
1. 是否有其他潜在问题
2. 是否符合最佳实践
3. 有哪些可以优化的地方
```

## 组合技巧

### 技巧1：明确阶段边界

```markdown
# 阶段一：[任务A]
[模板A的内容]
请先完成这个阶段，确认后再继续。

---

# 阶段二：[任务B]
基于阶段一的输出，[模板B的内容]
```

### 技巧2：输出接力

```markdown
上一步你分析得到了[结论]，现在请基于这个结论：
[下一步的要求]
```

### 技巧3：角色切换

```markdown
# 第一步：以[角色A]的身份
[任务A]

# 第二步：现在切换到[角色B]的身份
对上述输出进行[任务B]
```

### 技巧4：交叉验证

```markdown
# 第一步：设计方案
[方案设计]

# 第二步：批判性审视
现在假设你是一个保守的技术负责人，
你会对这个方案提出哪些质疑？

# 第三步：完善方案
基于上述质疑，修改和完善方案
```

## 常见错误避免

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 一次性堆砌所有要求 | 分阶段、分步骤提出 |
| 模板冲突（角色混乱） | 明确每个阶段的角色和要求 |
| 忽略阶段间的衔接 | 明确上一步输出如何用于下一步 |
| 输出格式不统一 | 在最终输出时统一格式要求 |
